96    /**
97     * Evaluates the given query using the Cypher query engine. The engine uses default morphism
98     * strategies, which is vertex homomorphism and edge isomorphism. The vertex and edge data of
99     * the data graph elements is attached to the resulting vertices.
100    * <p>
101    * Note, that this method used no statistics about the data graph which may result in bad
102    * runtime performance. Use {@link LogicalGraphOperators#query(String, GraphStatistics)} to
103    * provide statistics for the query planner.
104    * <p>
105    * In addition, the operator can be supplied with a construction pattern allowing the creation
106    * of new graph elements based on variable bindings of the match pattern. Consider the following
107    * example:
108    *
109    * <pre>
110    * <code>graph.query(
111    *  "MATCH (a:Author)-[:WROTE]->(:Paper)<-[:WROTE]-(b:Author) WHERE a <> b",
112    *  "(a)-[:CO_AUTHOR]->(b)")
113    * </code>
114    * </pre>
115    * <p>
116    * The query pattern is looking for pairs of authors that worked on the same paper. The
117    * construction pattern defines a new edge of type CO_AUTHOR between the two entities.
118    *
119    * @param query               Cypher query string
120    * @param constructionPattern Construction pattern
121    * @return graph collection containing the output of the construct pattern
122    */
123   GraphCollection query(String query, String constructionPattern);

259   /**
260    * Returns a subgraph of the logical graph which contains only those vertices
261    * and edges that fulfil the given vertex and edge filter function
262    * respectively.

265    * graph. Use {#toGellyGraph().subgraph()} for that behaviour.

269    * @return logical graph which fulfils the given predicates and is a subgraph

273     FilterFunction<EPGMEdge> edgeFilterFunction) {
